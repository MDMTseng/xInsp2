# multi_source_surge2 — FL r6 regression results

Sub-round of FL r6, run on a fresh agent that did NOT participate in
the original PR #22 work. Goal: verify the three PR #22 fixes hold via
a different driver path (different topology, different parallelism mix,
different test approach), and surface any new friction the fixes
themselves introduce.

VERDICT: **PASS** — all three PR #22 fixes hold. One new P2 surfaced
(see FRICTION.md), one happy meta-confirmation that Fix 3's docs help
in practice (see "Bonus" below).

## What this regression is

Topology deliberately different from `multi_source_surge/`:

- 2 sources, 1 detector (vs the original's 3+2)
- 5 sweeps in one driver run with varying N / queue / overflow,
  exercising counter-reset semantics across many start/stop cycles
- A dedicated FIX 2 sub-test: `set_watchdog_ms(500)` then `cmd:start`
  with N=4, capture the warn-log via the SDK's `on_log` callback —
  i.e. confirm the log actually reaches a `xinsp2.Client` consumer,
  not just stderr.
- A dedicated FIX 3 sub-test: compile a deliberately-shadowed
  `inspect_collision.cpp` and check what the developer sees when they
  hit the C2374.

## The three fixes

### FIX 1 — `dispatch_stats` reset semantics (P1-1)

**Doc check.** I read `docs/protocol.md`'s new `dispatch_stats` section
and the `cmd:start` reset note. I read the `writing-a-script.md`
parallelism section's "Don't subtract" callout. Then I wrote the driver
following only the docs (`per_run_stats_after()`: snapshot AFTER stop,
treat as per-run total, do NOT subtract a pre-snapshot).

The doc-driven driver got correct numbers across all 5 sweeps with no
underflow, no negative values, no off-by-one. The doc steers right.

**Empirical check.** 5 sweeps, varying queue/overflow:

| Sweep | N | q | overflow | active | drops | qmax/cap | sane |
|---|---|---|---|---|---|---|---|
| S1-N1-q32-clean | 1 | 32 | drop_oldest | 127 | 0 | 4/32 | yes |
| S2-N4-q64-light | 4 | 64 | drop_oldest | 159 | 0 | 29/64 | yes |
| S3-N1-q16-tight | 1 | 16 | drop_oldest | 137 | 73 | 16/16 | yes |
| S4-N4-q128-big | 4 | 128 | drop_oldest | 250 | 122 | 128/128 | yes |
| S5-N2-q32-newest | 2 | 32 | drop_newest | 148 | 220 | 32/32 | yes |

What we wanted to see and saw:
- **Reset takes effect.** S1 (no surge) reports qmax=4. S4 (200-frame
  surge) reports qmax=128. S1's small qmax did not bleed forward into
  later sweeps (sanity), and S4's big qmax did not bleed back into S5
  (S5's qmax=32 == its own cap, generated by its own surge).
- **Counter sign + magnitude sane.** No negative values, no values
  > queue_depth_cap. (`sane_counters=True` in every row.)
- **drop_newest vs drop_oldest discrimination intact.** S5
  (drop_newest under sustained surge) reports `dropped_newest=220`,
  `dropped_oldest=0`; S4 (drop_oldest) reports the inverse. Fix 1
  doesn't break this.

### FIX 2 — Watchdog warn-log under N>1 (P1-3)

**Test.** Set watchdog 500 ms via `cmd:set_watchdog_ms`, open a project
with `dispatch_threads=4`, `cmd:start`, capture all `log` messages via
`Client.on_log()`.

```
set_watchdog_ms -> {'ms': 500, 'trips': 0}
log lines during start: 1 total, 1 warn, 1 watchdog-related
  [warn] dispatch_threads=4 — script watchdog disabled under N>1
         (single-slot deadline state). Long-running ops should poll
         xi::cancellation_requested(). See docs/guides/writing-a-script.md.
```

**Negative control.** Disarm watchdog (`set_watchdog_ms 0`), `cmd:start`
again. Zero watchdog-related warns. The warn fires *only* when both
conditions (N>1 AND watchdog>0) hold — exactly the contract.

The warn arrives via the WS `log` channel (not just stderr) — it lands
in `Client._inbox_logs` and reaches the `on_log` callback. A driver
running with N=8 now has a programmatic signal that watchdog is
disabled, fixing the "switch from N=1 silently traded crash isolation"
problem PR #22 called out.

Minor note: the em-dash in the warn message comes through as a
mojibake byte sequence (`\xE2\x80\x94` rendered as garbled chars in
console under CP-950). See FRICTION P2 for the fix.

### FIX 3 — VAR(name, name) shadow doc cross-ref (P2-1)

**Bonus / meta-confirmation.** I hit this gotcha *while writing the
driver for the regression that tests it.* On my first attempt at
`inspect.cpp` I wrote:

```cpp
bool slow_path = (src_tag == TAG_B) && (slow_ms > 0);
VAR(slow_path, slow_path);
```

cl.exe fired:

```
inspect.cpp:92 error C2374: 'slow_path': redefinition; multiple init.
inspect.cpp:91 note: see declaration of 'slow_path'
inspect.cpp:92 error C2086: 'bool slow_path': redefinition
inspect.cpp:92 error C2371: 'slow_path': redefinition; different basic types
```

Without PR #22's cross-reference at `xi_var.hpp` line 205-211, I would
have stared at this and wondered why a single `VAR(...)` line caused
"redefinition" of a thing that was clearly only defined once. Because
the comment is now there, jumping to the macro definition (Ctrl-click
in any editor) puts the explanation right next to the implementation.
The natural pattern → diagnosis path is short enough that I figured it
out from the comment without needing to re-read the gotcha doc.

So Fix 3 worked on me, the test author. (Not great that the test
inspect.cpp had to be rewritten as the first regression-of-the-fix
data point, but it's the cleanest possible evidence the fix was
needed.)

**Explicit collision test.** `inspect_collision.cpp` deliberately
codes `VAR(foo, foo)`; `compile_and_load()` raises `ProtocolError`
whose `data.diagnostics` carries the C2374, C2086, C2371 chain. The
SDK's compile-error enrichment surfaces the file:line:code from cl.exe.
Two issues there, written up as P2 in FRICTION.md (mojibake on Chinese
locales — separate from the FL r2 `VSLANG=1033` workaround for the
backend's own emit path; the WS log warn from FIX 2 hits the same
class of bug).

## Per-source attribution under N=4

Every active inspect event in every sweep had `src ∈ {a, b}` — no
`unknown` and no "no_sources". FNV-1a tag routing is robust under N=4
across 800+ events in S4 alone. Same conclusion as the original FL r6
on a different topology.

## Latency observations

Latency rises sharply with queue saturation, as expected:
- S1 (clean): p95 ~17 ms
- S4 (saturated, 122 drops, qmax=128): p95 ~513 ms

This is the queue-time cost of `drop_oldest` under sustained
overload — events that *do* get dispatched have aged in the queue.
The 0.5 s p95 on S4 matches the back-of-envelope `q_depth_cap *
sleep_ms / N = 128 * 8 / 4 = 256 ms` plus surge-tail effects.

## Pass/fail

PASS:
- All three fixes verified.
- No new P0 / P1 introduced.
- One new P2 found (mojibake on warn payload + cl.exe diagnostics
  on Chinese locales) — written up.
